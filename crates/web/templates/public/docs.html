// quickstart

01 / Install the CLI

# macOS / Linux (Homebrew)
brew install liquidmetal/tap/flux

# Shell script (Linux / macOS)
curl -fsSL https://get.liquidmetal.dev | sh

# From source (requires Rust toolchain)
cargo install flux-cli

02 / Authenticate

flux login
# Opens browser for OIDC device flow
# If you have an invite code:
flux login --invite YOUR_CODE

flux whoami
# Shows your user info and active workspace

03 / Initialize a project

cd my-app
flux init
# Detects your language, asks for engine (metal/liquid)
# Writes liquid-metal.toml to your project root

# Or skip the prompts:
flux init --engine metal --name my-api

04 / Deploy

flux deploy
# Builds locally, uploads artifact, provisions service
# Returns a live URL when done

flux services
# List all services in your workspace

Auth

{# Deploy lifecycle #}

Deploy lifecycle

flux init Initialize a project. Detects language, writes liquid-metal.toml.
  --name <NAME> Override detected service name
  --engine <ENGINE> "metal" or "liquid" (prompts if omitted)
flux build Build locally without deploying
flux deploy Build, upload artifact, provision service. Returns a live URL.
  --skip-elf-check Skip ELF compatibility check (glibc vs musl detection)
flux releases <SERVICE> View deployment history for a service
flux rollback <SERVICE> Rollback to the previous deploy (or a specific deploy ID)
  --deploy_id <ID> Rollback to a specific deploy (default: previous)
  --yes, -y Skip confirmation prompt
{# Service management #}

Service management

flux services List all services in the active workspace
flux logs <SERVICE> View service logs
  --limit <N> Number of recent lines (default: 100)
  --follow, -f Follow log output (polls every 2s)
flux stop <SERVICE> Stop a running service
flux restart <SERVICE> Restart a service (stop + redeploy)
flux delete <SERVICE> Permanently delete a service. Cannot be undone.
  --yes, -y Skip confirmation prompt
flux open <SERVICE> Open service URL in your default browser

// engines

Metal vs Liquid

Metal Liquid
RuntimeFirecracker microVM (KVM)Wasmtime (WASI)
Targetx86_64-unknown-linux-muslwasm32-wasip1
ModelDedicated — always onServerless — scale to zero
Cold startNone (VM is always running)<10ms (cached module deserialize)
IsolationKVM + eBPF + cgroup + jailerWasm linear memory sandbox
NetworkingReal TCP/IP via TAP deviceHTTP only (WAGI per-request)
FilesystemYes — ext4 rootfsNo
BillingFixed monthlyPer invocation ($0.30/1M)
Best forAPIs, databases, long-running processesWebhooks, transforms, lightweight APIs

Use Metal when you need

  • • A real Linux kernel (syscalls, /proc, /dev)
  • • TCP/UDP networking (databases, game servers)
  • • Persistent filesystem storage
  • • Consistent latency with zero cold starts
  • • Binaries that use glibc or link C libraries

Use Liquid when you need

  • • Scale to zero (pay nothing when idle)
  • • Sub-millisecond cold starts
  • • Stateless request handlers (webhooks, APIs)
  • • Per-invocation billing for low-traffic services
  • • Maximum density (no VM overhead)

// configuration

liquid-metal.toml

Every project has a liquid-metal.toml in its root. Created by flux init.

Metal (Go example)

[service]
name = "my-api"
engine = "metal"
port = 8080

[build]
command = "go build -o app ."
output = "app"

Liquid (Rust example)

[service]
name = "webhook-handler"
engine = "liquid"

[build]
command = "cargo build --target wasm32-wasip1 --release"
output = "target/wasm32-wasip1/release/handler.wasm"

Fields

[service].name Service name (used as slug) required
[service].engine "metal" or "liquid" required
[service].port Port your binary listens on (Metal only) Metal only
[build].command Shell command to build your binary required
[build].output Path to the built artifact (binary or .wasm) required

// 05 — multi-service

Metal + Liquid

Deploy your API on Metal, your webhooks on Liquid, in the same project. Shared config, automatic service discovery.

{# ── The pattern ─────────────────────────────────────────────────── #}

The pattern

Metal — your API

Always-on. Connects to Postgres, Redis, anything with a socket. Handles auth, sessions, real-time features.

# my-api/liquid-metal.toml
[service]
name = "my-api"
engine = "metal"
port = 8080
tier = "one"

[build]
command = "cargo build --target x86_64-unknown-linux-musl --release"
output = "target/x86_64-unknown-linux-musl/release/my-api"

Liquid — your webhooks

Scales to zero. Wakes in under a millisecond. Handles Stripe webhooks, image processing, data transforms.

# my-webhooks/liquid-metal.toml
[service]
name = "my-webhooks"
engine = "liquid"

[build]
command = "cargo build --target wasm32-wasip1 --release"
output = "target/wasm32-wasip1/release/my-webhooks.wasm"
{# ── Shared env vars ─────────────────────────────────────────────── #}

Project-level env vars

Set an env var once at the project level. Every service in the project gets it.

# Set shared config for the whole project
$ flux env set --project my-project \
    DATABASE_URL=postgres://user:pass@db.internal:5432/mydb \
    STRIPE_SECRET_KEY=sk_live_...

# Both my-api (Metal) and my-webhooks (Liquid) receive these vars
# Service-level vars override project-level vars on collision
{# ── Service discovery ───────────────────────────────────────────── #}

Service discovery

When you deploy a service, every other running service in the same project is injected as an LM_SERVICE_* environment variable.

# Inside my-api, after deploying my-webhooks:
LM_SERVICE_MY_WEBHOOKS=https://my-webhooks.liquidmetal.dev

# Inside my-webhooks, after deploying my-api:
LM_SERVICE_MY_API=https://my-api.liquidmetal.dev

# Your code reads it like any other env var:
let webhook_url = std::env::var("LM_SERVICE_MY_WEBHOOKS")?;
client.post(&webhook_url).json(&payload).send().await?;

Peer URLs are updated on deploy and restart. Run flux restart <service> to pick up new peers.

{# ── Workflow ─────────────────────────────────────────────────────── #}

Workflow

# 1. Create project
$ cd my-api && flux init --engine metal
# This creates the project "my-api" and writes liquid-metal.toml

# 2. Deploy your API
$ flux deploy
# Live at https://my-api.liquidmetal.dev

# 3. Deploy your webhook handler (same project)
$ cd ../my-webhooks && flux init --engine liquid
$ flux deploy
# Live at https://my-webhooks.liquidmetal.dev

# 4. Set shared config
$ flux env set --project my-api DATABASE_URL=postgres://...

# 5. Restart both to pick up shared vars + peer discovery
$ flux restart my-api
$ flux restart my-webhooks

# Done. Both services share DATABASE_URL and know each other's URLs.
{# ── Priority ─────────────────────────────────────────────────────── #}

Env var priority

When the same key exists at multiple levels, the most specific wins:

1. LM_SERVICE_* — peer discovery (highest)

2. Service env vars — flux env set <service> KEY=VALUE

3. Project env vars — flux env set --project KEY=VALUE (lowest)

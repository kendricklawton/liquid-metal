## Liquid Metal: Engineering Protocol

### Product Thesis

Liquid Metal is a **PaaS (not IaaS)** delivering hardware-isolated compute with infrastructure transparency. We provide predictable performance (vCPU/RAM) without the "mystery resource pool" abstraction of Vercel or the management overhead of Kubernetes.

* **Metal**: Firecracker microVMs. Linux binaries, ext4 rootfs, TAP networking. ~100-250ms cold start.
* **Liquid**: Wasmtime/WASI execution. In-process, memory-only, no disk/TAP. <1ms startup.
* **Zero-Trust Infrastructure**: No K8s. No managed orchestrators. Bare metal or death.

---

### Tech Stack & Constraints

* **Languages**: Rust (Backend/Engine), Go (Web/CLI).
* **Web Stack**: Chi (Go), Templ, HTMX, Alpine.js (minimal). **No SPAs.**
* **Database**: PostgreSQL (Raw SQL via `tokio-postgres`, no ORMs).
* **Messaging**: NATS JetStream (The source of truth for provisioning).
* **Proxy**: Pingora (Cloudflare's framework) for edge routing.
* **Security**: Firecracker + Jailer + eBPF (Aya) for tenant isolation.
* **UI Aesthetic**: `rounded-none`. Strict utilitarianism.

---

### System Architecture & Data Flow

1. **CLI (`flux`)**: Reads `liquid-metal.toml`, uploads artifacts to S3, calls Rust API via ConnectRPC.
2. **Web (`chi`)**: Handles WorkOS OAuth. Proxies all data requests to the Rust API via ConnectRPC. **Never touches Postgres/NATS.**
3. **API (`axum/tonic`)**: The Brain. Writes to Postgres, publishes `ProvisionEvent` to NATS.
4. **Daemon**: NATS Consumer. Orchestrates Firecracker (Linux) or Wasmtime (Multi-platform).
5. **Proxy (`pingora`)**: Maps `slug -> upstream_addr` via DB lookup.

---

### Crate & Module Map

#### Rust Workspace (`/crates`)

* `common`: Shared `Engine` enums, `ProvisionEvent`, and config logic.
* `api`: Axum + ConnectRPC. Port `:7070`. Validates `X-Api-Key`.
* `proxy`: Pingora-based edge router.
* `daemon`: The worker. Firecracker/Wasmtime logic. Linux-only code gated via `#[cfg(target_os = "linux")]`.
* `ebpf-programs`: TC egress classifiers (Aya). *Excluded from workspace.*

#### Go Workspace (`go.work`)

* `web`: Dashboard (`/auth`, `/dashboard`). Uses Templ + HTMX.
* `cli`: Cobra-based tool (`flux`). Communicates directly with Rust API.
* `mcp`: MCP server. Exposes Liquid Metal operations as tools for Claude agents.
* `gen/go`: Shared Protobuf stubs generated by `buf`.

---

### Rules of the House

1. **The Lib/Main Split**: All Rust crates must use `lib.rs` for logic and `main.rs` for the entry point to ensure integration testability.
2. **Internal Communication**: Use ConnectRPC over h2c (HTTP/2 cleartext). No TLS between internal services.
3. **Configuration**: Zero hardcoded addresses. Use env vars (`NATS_URL`, `DATABASE_URL`, etc.).
4. **Deployment**: Binaries/Wasm modules are stored in S3-compatible storage using `uuid_v7` for immutable versioning.
5. **Compute Isolation**:
* **Serverless (Metal)**: 5-minute idle timeout.
* **Always-on (Metal)**: Continuous drain (Pro/Team only).
* **Liquid**: Per-invocation billing.

### MCP (Model Context Protocol) Integration

Liquid Metal exposes an **MCP server** so Claude and other AI agents can autonomously manage deployments without a human in the loop.

#### What It Is

MCP is Anthropic's open protocol for wiring AI agents to external tools. An MCP server exposes named tools (functions) that any MCP-compatible client (Claude Desktop, Claude Code, custom agents built with the Claude Agent SDK) can discover and call.

#### Architecture

The MCP server lives at `mcp/` — a standalone Go service that wraps the same ConnectRPC client used by the web dashboard.

```
Claude / Agent SDK client
    │ MCP (JSON-RPC over stdio or SSE)
    ▼
mcp/        Go MCP server — tool definitions + handlers
    │ ConnectRPC / h2c
    ▼
Rust API :7070
    │
    ▼
Postgres + NATS
```

The MCP server has **zero direct DB/NATS access** — same constraint as the web service. All data flows through the Rust API.

#### Exposed Tools

| Tool | Description |
|---|---|
| `list_services` | List all services in the active workspace |
| `deploy` | Build and deploy a service from a local directory |
| `get_logs` | Fetch build/runtime log lines for a service |
| `get_status` | Get current status + upstream address for a service |
| `list_workspaces` | List workspaces the authenticated user belongs to |
| `list_projects` | List projects in a workspace |

#### Anthropic SDK / Claude API Guidelines

When building agent features (autonomous deploy pipelines, log analysis, ops automation):

- **Default model**: `claude-sonnet-4-6` — latest and most capable
- **Model IDs**: Opus `claude-opus-4-6`, Sonnet `claude-sonnet-4-6`, Haiku `claude-haiku-4-5-20251001`
- **SDK**: Use the official Anthropic Go SDK (`github.com/anthropics/anthropic-sdk-go`) or Rust SDK (`anthropic-sdk-rust`) — never call the REST API directly
- **Auth**: `ANTHROPIC_API_KEY` env var — never hardcoded
- **Agent loops**: Use the Claude Agent SDK for multi-step autonomous tasks (deploy → verify → rollback on failure)

#### Rules

- MCP server authenticates to the Rust API with `X-Api-Key` (same as CLI) — token stored in env var `FLUX_API_KEY`
- No tool may perform destructive operations (delete service, deprovision) without an explicit `confirm: true` parameter
- All tool responses include structured JSON — no free-form text for machine consumers
- MCP server is opt-in for local dev: `task dev:mcp` starts it on stdio

### Development Commands

```bash
# Infrastructure
task up                   # Spin up Postgres, NATS, RustFS (S3 mock)
task metal:setup          # (Linux Root) Setup br0, jailer, cgroups

# Development
task dev:api              # Start Rust API
task dev:web              # Start Go Web (with Air reload)
task dev:daemon           # Start NATS consumer
task dev:mcp              # Start MCP server (stdio, for Claude Desktop / agent dev)

# Testing
cargo test --workspace    # Rust suite
cd web && go test ./...   # Go Web suite
cd cli && go test ./...   # CLI suite
